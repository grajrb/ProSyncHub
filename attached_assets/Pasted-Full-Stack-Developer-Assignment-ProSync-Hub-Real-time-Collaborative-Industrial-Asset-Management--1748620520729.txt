Full-Stack Developer Assignment: ProSync Hub - Real-time Collaborative Industrial Asset Management & Predictive Maintenance Platform

Objective:
Develop a secure, scalable, and real-time full-stack application, "ProSync Hub," for managing industrial assets, tracking their operational status, scheduling maintenance, predicting potential failures, and facilitating collaboration among industrial teams. The system should handle diverse data types, provide real-time updates, and be deployable to a cloud environment.

Tools and Technologies:

Frontend: JavaScript, React, Next.js, Redux

Backend: Node.js (Express.js), Socket.IO

Databases: PostgreSQL (Primary Relational), MongoDB (Document/Time-Series Data), Redis (Caching, Pub/Sub, Session Store)

DevOps & Cloud: Docker, Git, CI/CD (e.g., Azure DevOps Pipelines / GitHub Actions), Azure (AKS, Azure DB for PostgreSQL, Azure Cosmos DB for MongoDB API, Azure Cache for Redis, Azure Monitor)

Functional Requirements:

LLD Design (Low-Level Design):

DB Schema:

PostgreSQL:

Tables structure:

Users (user_id PK, username, password_hash, email, role_id FK, created_at, updated_at)

Roles (role_id PK, role_name)

Permissions (permission_id PK, permission_name)

RolePermissions (role_id FK, permission_id FK, PK(role_id, permission_id))

Assets (asset_id PK, asset_tag UNIQUE, name, description, model, manufacturer, serial_number, installation_date, location_id FK, asset_type_id FK, parent_asset_id FK NULL, documentation_url, qr_code_path, current_status, created_at, updated_at)

AssetTypes (asset_type_id PK, type_name, description)

Locations (location_id PK, location_name, description, plant_id FK)

Plants (plant_id PK, plant_name, address)

WorkOrders (work_order_id PK, asset_id FK, title, description, status (e.g., 'Open', 'In Progress', 'Completed', 'Cancelled'), priority, type (e.g., 'Preventive', 'Corrective'), assigned_to_user_id FK, reported_by_user_id FK, scheduled_date, completion_date, created_at, updated_at)

MaintenanceSchedules (schedule_id PK, asset_id FK, task_description, frequency (e.g., 'Daily', 'Weekly', 'Monthly', 'Custom_Days'), next_due_date, last_performed_date, associated_work_order_template_id FK NULL)

SpareParts (part_id PK, part_number, name, description, quantity_on_hand, reorder_level, location_id FK)

WorkOrderParts (work_order_id FK, part_id FK, quantity_used, PK(work_order_id, part_id))

Foreign Keys: As indicated by FK above, ensuring referential integrity.

Normalization of data: Aim for 3NF for relational data where appropriate to reduce redundancy and improve data integrity.

MongoDB:

Collections:

AssetSensorReadings: { asset_id (indexed), timestamp (indexed), sensor_type, value, unit, raw_data_object } (Time-series like data)

AssetEventLogs: { asset_id (indexed), timestamp, event_type (e.g., 'ERROR', 'WARNING', 'INFO'), message, details_object }

UserActivityFeeds: { user_id (indexed), timestamp, activity_type, description, related_entity_type, related_entity_id }

ChatMessages: { room_id (e.g., work_order_id, asset_id), sender_user_id, timestamp, message_content, attachments_array }

MaintenanceChecklists: { checklist_id, work_order_id FK (optional), name, items: [{text, is_checked, checked_by_user_id, checked_at}]} (Flexible structure)

Classes (Conceptual for Backend - Node.js/Express Services & Frontend - React Components):

Backend (Node.js/Express.js - Services/Controllers/Models):

AuthService: Handles user registration, login, token generation/validation, session management.

Attributes: userModel, roleModel.

Functions: registerUser(), loginUser(), validateToken(), assignRole().

AssetService: Manages CRUD operations for assets, types, locations.

Attributes: assetModel, assetTypeModel, locationModel.

Functions: createAsset(), getAssetById(), updateAssetDetails(), getAssetsByLocation().

WorkOrderService: Manages work orders, assignments, status updates.

Attributes: workOrderModel, userService (dependency), notificationService (dependency).

Functions: createWorkOrder(), assignWorkOrder(), updateWorkOrderStatus(), getWorkOrdersForAsset().

MaintenanceService: Handles maintenance schedules and predictive logic.

Attributes: maintenanceScheduleModel, assetSensorReadingModel.

Functions: createSchedule(), checkForDueMaintenance(), predictFailure().

RealtimeService (Socket.IO Handler): Manages WebSocket connections and broadcasts.

Attributes: io_instance.

Functions: broadcastAssetUpdate(), sendNotificationToUser(), handleChatMessage().

DataIngestionService (for sensor data - can be separate worker):

Attributes: assetSensorReadingModel (MongoDB), redisClient.

Functions: processSensorData(), triggerAlerts().

Frontend (React/Next.js - Components/Redux State):

AssetDashboard: Displays overview of asset statuses, KPIs.

Attributes (State/Props): assetList, alertCount, filterOptions.

Functions: fetchDashboardData(), handleFilterChange().

AssetDetailView: Shows detailed information for a single asset, including sensor data charts, maintenance history.

Attributes (State/Props): selectedAsset, sensorDataHistory, workOrderList.

Functions: fetchAssetDetails(), subscribeToAssetUpdates().

WorkOrderForm: For creating/editing work orders.

Attributes (State/Props): workOrderData, userListForAssignment.

Functions: handleSubmit(), handleInputChange().

NotificationComponent: Displays real-time notifications.

Attributes (State/Props): notificationsList.

Functions: markAsRead().

Redux Slices/Reducers: authSlice, assetSlice, workOrderSlice, notificationSlice.

Interaction between two or more classes/modules:

WorkOrderService calls NotificationService when a work order is assigned.

RealtimeService listens for events (e.g., from DataIngestionService via Redis Pub/Sub or direct call) and broadcasts to relevant frontend clients.

Frontend AssetDetailView subscribes to real-time updates for a specific asset via RealtimeService (Socket.IO).

AssetService interacts with AssetModel (PostgreSQL) for persistence. DataIngestionService interacts with AssetSensorReadingModel (MongoDB).

OPEN-CLOSE Relationships (Design for Extensibility):

New sensor types can be added without modifying core DataIngestionService; parsing/handling logic can be modular.

New predictive maintenance algorithms can be plugged into MaintenanceService if designed with a strategy pattern.

Reporting modules can be added independently, consuming data from existing services/DBs.

Role-based access control allows adding new roles/permissions without code changes to core feature logic, only configuration.

Handling of Dependencies:

Backend: npm for package management. Dependency Injection (DI) principles for services (e.g., passing service instances or using a DI container).

Frontend: npm or yarn. Redux for state management, React Context for simpler shared state.

1. Asset Registry & Hierarchy:
* Objective: Securely store and manage detailed information about industrial assets in a hierarchical structure.
* Technologies: PostgreSQL for structured asset data; Next.js/React for UI.
* Features:
* CRUD operations for assets, types, locations, plants.
* Define parent-child relationships between assets.
* Store documents (manuals, specs - URL to S3/Azure Blob if large, or directly if small).
* QR code generation (link to asset detail page).

2. Real-time IoT Data Ingestion & Monitoring:
* Objective: Ingest (or simulate) and display real-time operational data from assets.
* Technologies: Node.js/Express for data ingestion endpoint (or direct Socket.IO for simulation), Socket.IO for real-time broadcast, MongoDB for storing time-series sensor data, Redis for temporary/hot data or pub/sub, React/Next.js for dashboards.
* Features:
* API endpoint or simulated stream for sensor data (temperature, vibration, pressure, uptime, error codes).
* Live dashboards with customizable widgets showing current asset status and KPIs.
* Historical data visualization (charts, graphs) from MongoDB.

3. Predictive Maintenance Module:
* Objective: Analyze asset data to predict potential failures and suggest maintenance.
* Technologies: Node.js for rule-based logic; PostgreSQL/MongoDB for historical data. (Advanced: could be a separate Python microservice for ML).
* Features:
* Configurable rule-based alerts (e.g., "vibration > X for Y minutes").
* (Basic) Trend analysis on historical sensor data to flag anomalies.
* Generate proactive maintenance recommendations.

4. Maintenance Scheduling & Work Order Management:
* Objective: Create, assign, track, and manage maintenance work orders and schedules.
* Technologies: PostgreSQL for work orders and schedules; React/Next.js for UI; Node.js/Express for backend logic.
* Features:
* CRUD for work orders (preventive, corrective).
* Assign technicians, track status, log completion details.
* Calendar view for scheduled maintenance.
* Link work orders to spare parts inventory.

5. Real-time Collaboration & Notifications:
* Objective: Enable real-time communication and notifications for teams.
* Technologies: Socket.IO for real-time messaging/notifications; Node.js/Express for backend; React/Next.js for UI; Redis for Pub/Sub for distributed notifications.
* Features:
* In-app notifications for critical alerts, new work orders, status changes.
* (Optional) Simple chat functionality associated with work orders or assets.
* Activity feed per asset/work order.

6. User Roles & Permissions:
* Objective: Implement a robust access control system.
* Technologies: PostgreSQL for user/role/permission storage; Node.js/Express for middleware-based authorization.
* Features:
* Define roles (Admin, Supervisor, Technician, Operator).
* Assign permissions to roles for accessing different features/data.

7. Reporting & Analytics:
* Objective: Provide insights into asset performance and maintenance operations.
* Technologies: PostgreSQL/MongoDB for data aggregation; Node.js/Express for API endpoints; React/Next.js with charting libraries for visualization.
* Features:
* Reports on asset uptime, MTBF, MTTR, maintenance costs.
* Customizable dashboards for key performance indicators.

Non-Functional Requirements:

Usability: Intuitive and responsive UI/UX optimized for both desktop and potentially tablet use in industrial settings.

Performance: Fast loading dashboards, real-time updates with minimal latency, efficient database queries for large datasets.

Security: Secure user authentication (e.g., JWT with refresh tokens), protection against common web vulnerabilities (XSS, CSRF, SQLi), encrypted data in transit (HTTPS) and at rest (if required).

Scalability and Reliability: System designed to handle an increasing number of assets, users, and data points. High availability for critical monitoring and operational functions. Fault tolerance in data ingestion and processing.

Maintainability: Well-structured, documented, and testable code. Modular design to allow for easier updates and feature additions.
